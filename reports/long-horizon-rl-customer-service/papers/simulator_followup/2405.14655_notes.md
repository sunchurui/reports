# 2405.14655 — Multi-turn Reinforcement Learning from Preference Human Feedback

**论文**：Multi-turn Reinforcement Learning from Preference Human Feedback（缩写 **MTPO**）
**作者**：Lior Shani, Aviv Rosenberg, Asaf Cassel（共同一作）+ Oran Lang, Daniele Calandriello, Bilal Piot, Avital Zipori, Hila Noga, Orgad Keller, Idan Szpektor, Avinatan Hassidim, Yossi Matias, Rémi Munos
**单位**：Google Research + Google DeepMind + Tel Aviv University
**发表**：**NeurIPS 2024** · arXiv 2405.14655（v2: 2024-12）

## 一句话总结

把 RLHF 从"单轮偏好"扩展到"两段完整对话之间的偏好"，给出有 Nash 收敛保证的 mirror-descent 算法 (MTPO) 并在 Education Dialogue 任务上验证。

## 论文要解决的问题

**主流 RLHF 的根本局限**：所有现有 RLHF/DPO/Nash-MD 算法都建模为单轮 contextual bandit——给一个 prompt → 模型出一个 response → 拿一个偏好反馈。

**多轮场景下这种建模直接失效**，论文给的核心论证（Section 3 末段）：

> 一个聊天机器人**直接回答**用户的提问通常是好的——除非它信息不足，这时**反问澄清**才是更好的动作。但一个标注员**没法在不看后续对话的情况下**判断"直接答 vs 反问"哪个好。所以**逐轮打偏好标签会引入破坏性偏差**——动作的好坏取决于后续怎么发展，单看当前轮无法评估。

这是论文最核心的洞察——它否定了"把多轮拆成多个单轮分别 RLHF"这种朴素扩展。

## 形式化建模：Contextual MDP + 末态偏好

**与 RLHF 的差异**：

| | 单轮 RLHF (Christiano 2017 / Ouyang 2022) | Multi-turn RLHF (本文) |
|---|---|---|
| 建模 | Contextual Multi-Armed Bandit (CMAB) | **Contextual MDP (CMDP)** with horizon H |
| 偏好对象 | 两条 response | **两条完整 trajectory（直到末态 x_{H+1}）** |
| Bradley-Terry | 偏好 = sigmoid(reward 差) | **不假设有底层 reward**，直接学偏好 |
| 优化目标 | argmax E[reward] | **argmax min P(π ≻ π')**（Nash equilibrium） |

**核心设定**：

- 状态空间 X 按 horizon 切分：X = X_1 ⊍ X_2 ⊍ ... ⊍ X_{H+1}
- 在多轮对话里，state x_h 是"到第 h 轮为止的完整对话历史"，action y_h 是 agent 第 h 轮生成的 utterance
- 环境（=人类用户）的反应：next_state = concat(x_h, y_h, user_response)
- **偏好只在末态给**：P(x_{H+1} ≻ x'_{H+1} | c)，对比的是两条完整对话的最终状态
- 偏好对策略的扩展：P(π ≻ π') = E_{两个轨迹分别由 π 和 π' 生成}[P(x_{H+1} ≻ x'_{H+1})]

**目标**：找一个对所有其他策略都"偏好胜出"的 π*，即 Nash equilibrium。论文证明（Lemma 3.2）regularized preference model 下 Nash 唯一存在。

## 核心算法：MTPO（Multi-turn Preference Optimization）

整体思路是把单轮的 **Nash-MD**（Munos et al. 2023）扩展到多轮。

### MTPO-α（α-regularized preference model）

定义 α-regularized 偏好：

```
P_α(π ≻ π') = P(π ≻ π') - α · KL_p(π || μ) + α · KL_p(π' || μ)
```

其中 μ 是 reference policy（一般是 SFT 模型），KL_p 是策略在 trajectory 分布上的 KL 散度。

**关键引理 3.1**（KL 分解）：trajectory-level KL 等于各状态上 turn-level KL 的期望和：

```
KL_p(π || π') = E_{π,p}[Σ_h KL(π || π')[x_h]]
```

这条引理是后续算法可解的关键——它把"对整段对话的偏好优化"分解成了"逐 state 优化"。

### Preference-based Q-function

定义关键的 preference-based Q：

```
Q^{π,π'}_α(x_h, y_h) = E_{π,p}[P(x_{H+1} ≻ π') - α · Σ_{h'≥h} KL(π||μ)[x_{h'}] | x_h, y_h]
```

直观理解：在状态 x_h 采取动作 y_h、之后按 π 走完，对手按 π' 走，能得到的 regularized 偏好回报的期望。注意它**没有 reward**，但有"末态偏好"作为 trajectory 终点的回报。

### MTPO 更新规则

每一轮迭代 t，用 mirror descent 更新：

```
π_{t+1}(· | x_h) = argmax_π { η_t · E_y~π [Q^{π_t,π_t}_α(x_h, y)] - KL(π || π_t^α)[x_h] }
```

其中 π_t^α(· | x_h) ∝ π_t(· | x_h)^{1-αη_t} · μ(· | x_h)^{αη_t} 是策略和参考的几何插值。

**收敛保证**：论文 Theorem 4.1 证明 MTPO 以 O(1/T) 速率收敛到 Nash 均衡。

### 实用变体：MTPO²

MTPO 需要每次迭代估计 Q^{π_t,π_t}，实操较贵。MTPO² 是简化版——直接采两条轨迹两两对比，类似 self-play DPO。论文实验主要用 MTPO²。

## 实验设置

### Education Dialogue（论文专门搭的环境）

teacher agent 教 student agent 一个随机话题。state = 到当前为止的对话历史 + 话题。trajectory 末态由另一个 LLM judge 判断"哪段教学过程更好"——形成偏好反馈。

**为什么搭这个环境**：现有多轮对话基准（MultiWOZ 等）都是 task-oriented + 有标准答案，无法自然产生"trajectory-level 偏好"。Education Dialogue 没有标准答案，必须看整段对话才能判断好坏。

**数据集已开源**：[google-research-datasets/Education-Dialogue-Dataset](https://github.com/google-research-datasets/Education-Dialogue-Dataset)

### Car Dealer（验证算法 reward 等价性）

这个环境有显式 reward（成交价 - 成本）。实验目的：证明 MTPO 在**只用偏好信号**的情况下能逼近"用真实 reward 做 RL"的 baseline。结果：MTPO² 与 reward-based PPO 表现持平。

### 关键实验结论

1. **vs single-turn RLHF**：在 Education Dialogue 上，MTPO² 显著优于 PPO+BT-style RLHF（因为后者把多轮硬拆成单轮，丢失 trajectory 信号）
2. **vs Nash-MD（单轮版）**：MTPO² 在多轮任务上优于把 Nash-MD 用在每个单轮的方案
3. **数据效率**：在 Car Dealer 上 MTPO² 用偏好信号达到与 reward-based RL 相当的性能——证明偏好信号即使更弱，但在多轮 setting 下足够

## 与 CollabLLM 的本质区别

两篇都做"多轮 RL"，但思路完全不同：

| 维度 | MTPO (Shani, NeurIPS 2024) | CollabLLM (Wu, 2025-02) |
|---|---|---|
| 反馈来源 | **真人对比两段完整对话** | **LLM 模拟用户继续聊 w 轮** |
| 核心机制 | Nash 均衡 + mirror descent | Monte Carlo rollout + reward shaping |
| 是否需要 user simulator | ❌ 不需要 | ✅ 强依赖（且是软肋） |
| 是否假设有 reward | ❌ 直接学偏好 | ✅ 需要 conversation-level reward 函数 |
| 理论保证 | ✅ 强（Nash 收敛） | ❌ 弱（实证为主） |
| 工业标注成本 | 高（多轮偏好对比） | 低（只需要 reward signal） |

**两者是不同象限的方法**，各有适用场景：
- MTPO 适合"有真人偏好数据但 reward 难定义"的场景（如客服满意度）
- CollabLLM 适合"有 task reward 但想榨干多轮收益"的场景（如代码 / 数学）

## 对客服 RL 训练的可借鉴度 ⭐

**结论：可借鉴度比 CollabLLM 更高，因为绕开了 user simulator 这个软肋。**

### 1. 客服场景天然适合 trajectory-level 偏好

论文 Section 3 给出的"该不该反问澄清"的论证完全是客服场景的写照：

- 用户说"我的单怎么还没到"
- bot 选项 A：直接给当前位置 + ETA → 单轮看高效
- bot 选项 B：先问"是不是着急？要不要联系骑手？" → 单轮看啰嗦
- **谁更好？看后续 4-5 轮发展**——A 后续可能要返工 3 轮（用户其实想催骑手），B 一句话省 3 轮

**这正是 MTPO 设计动机的字面情况**——逐轮标偏好会标错，trajectory-level 标才能反映真实质量。

### 2. 标注成本可控

- 不需要 user simulator（绕开 CollabLLM 的核心软肋）
- 不需要定义"问题是否解决"的客观函数（绕开 task reward 难定义）
- 只需要让客服 supervisor 在两段历史对话里挑"哪段处理得更好"——这是客服 QA 团队**本来就在做的事**

### 3. 实操路径

1. 抽 1000-2000 段历史多轮对话作 base
2. 配对成 500-1000 组对比，让客服 supervisor 标偏好（每组 1-2 分钟，总成本 20-40 工时）
3. 训 trajectory-level reward model（输入是整段对话）
4. 用 MTPO² 算法（开源参考实现可找 google-research）做 RL 微调
5. 在你的 src/sim/ 仿真器上跑 evaluation

### 4. 局限

- 论文实验在 Education Dialogue（小规模 LLM judge 偏好）上做，**没在工业级长尾数据上测**
- Nash 收敛是理论保证，**实际深度 RL 实现里收敛速度未必如理论那么快**
- 标注一致性问题：让标注员对比"两段对话哪段好"，对客服领域知识要求高，新手标不出来
